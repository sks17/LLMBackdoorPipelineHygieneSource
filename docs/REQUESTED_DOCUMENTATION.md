## Specific questions

1. **Models.** Which exact HF model IDs and revisions are approved? Any local checkpoints on the
   cluster to point at instead of downloading? Confirm `trust_remote_code` per model.
2. **Tokenizer pinning.** Pin tokenizer versions for reproducibility? Token boundaries affect
   truncation, so a fixed `transformers`/`tokenizers` version matters.
3. **Chat templates.** Use each tokenizer's built-in template, or supply explicit per-model
   templates? Append a generation prompt for the survival (non-generation) phase?
4. **Memory policy fidelity.** Should `keep_recent_messages` match LangChain `trim_messages`
   semantics exactly (token vs message strategy, `include_system`, partial-message handling)?
5. **Summarizer.** Which model/prompt produces summaries for semantic-survival study? Run on the
   cluster or via an API? (Currently a deterministic placeholder; semantic survival deferred.)
6. **Cluster.** Klone or Tillicum (or both)? Account, partition/QoS, max walltime, GPU type,
   storage quota and path (`/gscratch/<group>` vs `/gpfs/projects/<group>`)?
7. **Scale.** Target trial count for the first run (pilot ≈ 10k vs blog ≈ 180k)? Shard size?
8. **Outputs.** JSONL only, or Parquet for survival/generation results? Keep raw final-prompt text
   for all trials or a sample (default: sample via `--log-prompts`)?
9. **Triggers.** Confirm the canary set (random, natural phrase, multi-token, boundary). Any
   unicode/normalization cases to include now?
10. **Base-conversation generation.** *Implemented (Task 09; extended 2026-07-02 per
    `docs/DEEP_RESEARCH_FEEDBACK.md`).* The synthetic arm is generated by
    `src/trigger_audit/generation/conversation_generator.py`, a peer of the dataset adapter: it
    produces raw role-tagged content from deterministic seeds and hands it to the *same*
    `to_base_conversation` (length-bin + slot-plant), so synthetic and real bases differ only in
    content. Backends: `MockBackend` (deterministic, offline), `OllamaBackend` (local Qwen via Ollama;
    `think=False`, tolerant JSON parsing, anti-repetition decoding), and **`AgentAuthoredBackend`**
    (Claude-authored conversations served from a JSONL — the strong-generator path, since we have **no
    hosted-API access**) are live; `TransformersBackend` (cluster GPU; **recommended fill: Pythia**,
    for reproducibility) and `ApiBackend` are documented stubs. Each base now records **rich
    provenance** — `generation_model`, `prompt_template_version`, `generation_params` (decoding
    config), `language`, and the balanced `persona` covariate — and `materialize_synthetic_corpus`
    writes a `<output>.report.json` generation report (backend, template version, fallback rate,
    exact-duplicate count). A **quality gate** in `validate_generated` rejects degenerate output
    (gross turn over-production or whole-conversation repetition), so small-model degeneracy falls
    back to the deterministic mock backend (recorded honestly) instead of entering the corpus.
    **Still needed:** the approved cluster model id/revision (Pythia) + GPU allocation + `torch` for
    `TransformersBackend`. Note: the diverse, high-quality bulk of the corpus is expected from
    agent-authored (Claude) content, with local Qwen as a style-diversity source.

## Dataset ingestion for the H4 (`existing_dataset`) arm — Task 07

The `existing_dataset` arm (real corpora, for the H4 real-vs-synthetic comparison) is ingested by
`src/trigger_audit/io/dataset_adapter.py`. The adapter *structure* is complete and validated with a
synthetic `MockChatParser`; the per-source parsers are **blocked** on each source's real record
format and usage terms and ship as documented `NotImplementedError` stubs (`LMSYSParser`,
`WildChatParser`, `LongDocParser`) so no JSON shape is guessed.
